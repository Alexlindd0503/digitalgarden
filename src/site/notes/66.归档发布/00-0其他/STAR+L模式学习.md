#面试 #方法论

```ad-summary
title: 总结

- STAR+L 是面试讲项目经历的结构化框架，比流水账更有说服力
- S 点矛盾、T 明目标、A 讲方案、R 给数据、L 说成长
- 核心是"证明自己"，不是"介绍项目"
- 数据和反思是拉开差距的两个关键点
```

## 1. 框架结构

| 字母 | 含义 | 要点 |
|---|---|---|
| S（Situation） | 背景 | 一句话点出矛盾或痛点，不要铺垫太长 |
| T（Task） | 任务 | 明确你个人要解决的目标，突出"我的职责" |
| A（Action） | 行动 | 分步骤讲怎么做，体现技术思考和决策过程 |
| R（Result） | 结果 | 用数据和业务价值收尾，量化影响 |
| L（Learnings） | 学习 | 说成长与反思，体现深度思考 |

## 2. 各环节怎么讲

### 2.1 S - 背景，一句话点矛盾

不要从"我们公司是做什么的"开始，直接说问题是什么。

> 百万 DAU，大 V 一发帖要写上万条 Feed，延迟 3 分钟，用户疯狂吐槽。

背景够了，面试官已经知道这是个什么量级的问题。

### 2.2 T - 任务，说清你的角色

强调"我"而不是"我们"，让面试官知道你具体负责什么。

> 作为核心开发，我的目标是把写入延迟降到秒级，同时控制 Redis 的资源消耗不继续膨胀。

### 2.3 A - 行动，这是最重要的部分

分步骤讲，体现你的技术判断和取舍，而不是只说"我用了 Kafka"。

好的 A 要回答：
- 为什么选这个方案，而不是其他方案？
- 遇到了什么难点，怎么解决的？
- 有没有做过方案对比？

示例：
> 1. 引入 Kafka 削峰填谷，把同步写入改成异步，先解决阻塞问题
> 2. 推拉结合策略：普通用户走推模式，大 V（粉丝超过阈值）走拉模式，写入量降了 80%
> 3. 多级缓存 + Canal 订阅 Binlog 保证缓存一致性，避免直接穿透数据库

### 2.4 R - 结果，必须有数据

没有数据的结果等于没说。尽量量化：

- 性能指标：延迟从 3 分钟降到 5 秒
- 资源成本：Redis 从 20 台缩到 10 台，节约 50%
- 业务影响：用户留存涨 15%

如果真的没有精确数据，也要给出量级估算，比如"大约提升了 X 倍"。

### 2.5 L - 学习，拉开差距的地方

大多数人讲到 R 就停了，L 是区分普通候选人和优秀候选人的地方。

说什么：
- 这个方案有什么局限性，下次会怎么改？
- 参考了哪些业界方案，有什么新认知？
- 这个经历改变了你对某个技术/架构的理解？

示例：
> 参考了 Twitter 的 Fanout 和微博的混合推拉模式后，意识到架构没有银弹，阈值的设定本身就是个需要持续调优的业务问题，不是一次性的技术决策。

## 3. 完整示例：Feed 流项目

**S**：百万 DAU，大 V 一发帖要写上万条 Feed，延迟 3 分钟，用户疯狂吐槽。

**T**：作为核心开发，目标是把写入延迟降到秒级，且不让 Redis 资源继续膨胀。

**A**：
1. 引入 Kafka 削峰填谷，同步写入改异步
2. 推拉结合，大 V 超过粉丝阈值走拉模式，写入量降 80%
3. 多级缓存 + Canal 订阅 Binlog，保证缓存一致性

**R**：延迟从 3 分钟降到 5 秒，Redis 从 20 台缩到 10 台，成本节约 50%，用户留存涨 15%。

**L**：参考 Twitter、微博的方案后，对混合推拉模式理解更深，也意识到架构取舍没有银弹，阈值本身是个需要持续调优的业务问题。

## 4. 常见误区

- **背简历**：把项目介绍一遍，没有矛盾和解决过程，面试官听不进去
- **流水账**：罗列做了什么，没有聚焦核心问题和方案选择
- **空口吹**：说"性能大幅提升"，没有数据支撑，可信度低
- **止步于结果**：不说 L，显得只会执行，缺乏深度思考

面试不是比项目大，而是比谁能更好地证明自己解决问题的能力。
