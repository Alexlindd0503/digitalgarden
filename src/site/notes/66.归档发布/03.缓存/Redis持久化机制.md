#redis #缓存 #持久化

## 1. 两种持久化方式

Redis 有两种持久化：RDB 保存某个时刻的全量快照，AOF 记录每条写命令。生产环境推荐开启混合持久化，结合两者优点。

| | RDB | AOF |
|--|-----|-----|
| 原理 | 全量数据快照 | 追加写命令日志 |
| 恢复速度 | 快 | 慢（需重放命令）|
| 数据丢失 | 两次快照之间的数据 | 最多丢 1 秒（everysec）|
| 文件大小 | 小 | 大（可重写压缩）|

## 2. AOF 怎么工作的？

AOF 是写后日志，先执行命令写入内存，再记录日志。好处是不会因为日志写失败影响当前命令，坏处是命令成功但还没来得及写日志时挂了，这条数据就丢了。

三种刷盘策略：

- `always`：每条命令都同步刷盘，不丢数据，性能最差
- `everysec`：每秒刷一次，最多丢 1 秒数据，推荐
- `no`：交给操作系统决定，性能最好，可能丢较多数据

```conf
appendonly yes
appendfsync everysec
```

### 2.1 AOF 重写

AOF 文件会越来越大，重写机制用来压缩它。原理是读取当前数据库状态，为每个 key 生成一条等效命令写入新文件，多条历史命令合并成一条。

重写由后台子进程完成，不阻塞主线程。重写期间新来的写命令会同时写入原 AOF 文件和重写缓冲区，重写完成后合并，保证不丢数据。

```conf
auto-aof-rewrite-min-size 64mb    # 文件超过 64MB 才触发
auto-aof-rewrite-percentage 100   # 且是上次重写后的 2 倍
```

## 3. RDB 怎么工作的？

`bgsave` 命令 fork 一个子进程在后台生成快照，主线程继续处理请求。子进程和主线程共享内存，主线程修改数据时会触发写时复制（COW），复制一份副本给子进程读，保证快照数据一致。

内存越大，fork 操作耗时越长，这段时间主线程会短暂阻塞。所以要避免 bigkey，减少 fork 的开销。

```conf
# 自动触发 bgsave 的条件
save 900 1      # 900 秒内至少 1 个 key 变化
save 300 10     # 300 秒内至少 10 个 key 变化
save 60 10000   # 60 秒内至少 10000 个 key 变化
```

## 4. 混合持久化（推荐）

混合持久化开启后，AOF 重写时会先写一个 RDB 快照，再追加增量 AOF 命令。恢复时先加载 RDB（快），再重放少量 AOF 增量（补全数据），比纯 AOF 恢复快很多。

```conf
aof-use-rdb-preamble yes
appendfsync everysec
```

## 5. 怎么选？

- 纯缓存场景（数据可以从 DB 重建）：关掉持久化，性能最高
- 一般业务：混合持久化 + `everysec`，平衡性能和可靠性
- 不能丢数据：AOF + `always`，性能会下降明显
