#redis #集群 #高可用

```ad-summary
title: 总结

- 小规模用主从 + 哨兵，数据量大或 QPS 高上 Cluster
- 哨兵通过 Raft 投票选 Leader 执行故障转移，选新主库按优先级 → 复制进度 → 运行 ID 打分
- 脑裂问题用 `min-slaves-to-write` + `min-slaves-max-lag` 限制主库写入条件
- 主从复制断线重连优先走增量同步，backlog 追不上才全量
```

## 1. 三种部署方式怎么选？

| | 主从复制 | 哨兵模式 | Cluster |
|--|---------|---------|---------|
| 高可用 | 无自动故障转移 | 自动故障转移 | 自动故障转移 |
| 数据分片 | 否 | 否 | 是 |
| 横向扩展 | 否 | 否 | 是 |
| 适合数据量 | < 10GB | < 10GB | 不限 |
| 适合 QPS | < 10 万 | < 10 万 | 不限 |
| 复杂度 | 低 | 中 | 高 |

小规模用主从 + 哨兵，数据量大或 QPS 高就上 Cluster。

## 2. 主从复制

### 2.1 全量复制

从库第一次连接主库，或者断线太久 backlog 追不上，触发全量复制：

```
从库发 psync → 主库执行 bgsave 生成 RDB → 发送 RDB → 从库加载
→ 主库把这期间的增量命令（replication buffer）也发过去
```

### 2.2 增量复制

正常同步时，主库把写命令写入 `repl_backlog_buffer`（环形缓冲区），从库持续读取执行。断线重连后，只要偏移量还在 backlog 范围内，就走增量同步，不用全量。

```conf
repl-backlog-size 1mb  # 根据写入速度 × 预估断线时间 × 2 来设置
```

### 2.3 主从从模式

从库多了，主库生成和传输 RDB 的压力会很大。可以让从库再挂从库，分散压力：

```
主库 → 从库1 → 从库2、从库3
```

## 3. 哨兵模式

哨兵负责三件事：监控主从库是否在线、主库挂了时选出新主库、通知从库和客户端切换连接。

### 3.1 哨兵集群怎么组建？

哨兵之间通过 pub/sub 互相发现。每个哨兵连上主库后，订阅 `sentinel:hello` 频道，定期发布自己的信息，其他哨兵收到后建立连接，自动组成集群。

哨兵通过向主库发 `INFO` 命令获取从库列表，然后也去连接所有从库。

### 3.2 怎么选新主库？

先筛掉不健康的从库（下线、网络不稳定），再按优先级打分：

1. `slave-priority` 配置的优先级，越小越优先
2. 复制进度（`slave_repl_offset`），越接近主库越优先
3. 运行 ID，ID 小的优先

选出候选后，哨兵基于 Raft 投票，获得多数票（N/2+1）的哨兵成为 Leader，由它执行故障转移。

### 3.3 脑裂问题

主库网络抖动，哨兵判定它下线并选出新主库，但原主库其实还活着，这时候出现两个主库，客户端还在往原主库写数据，等原主库重新加入集群变成从库时，这些数据就丢了。

解决办法是限制主库的写入条件：

```conf
min-slaves-to-write 1       # 至少 1 个从库在线才允许写入
min-slaves-max-lag 12       # 从库延迟不超过 12 秒
down-after-milliseconds 10000  # 哨兵判定下线的超时时间
```

这样主库网络断开后，10 秒内哨兵判定下线，12 秒内没有从库同步，主库就会拒绝写入，避免脑裂期间的数据丢失。

## 4. Redis Cluster

Cluster 通过哈希槽实现数据分片，16384 个槽分配给各节点，每个 key 通过 `CRC16(key) % 16384` 定位槽，再找到对应节点。每个主节点可以挂从节点，主节点挂了从节点自动提升。

槽分配、重定向（MOVED/ASK）、槽迁移、Hash Tag 详见 [[66.归档发布/03.缓存/Redis Cluster槽管理机制\|Redis Cluster槽管理机制]]。

Cluster 的限制：
- 不支持跨槽多键操作（除非用 Hash Tag）
- 只能用 db 0
- 客户端需要支持重定向逻辑
